Oral de presentation - Partie backend

Cut&Go

Presentation simple de la partie invisible du site : ce qui gere les comptes, les salons, les rendez-vous et les informations affichees a l'utilisateur.

Objectif : 3 minutes

Introduction

Dans cette partie, je vais presenter ce que fait le backend. C'est la partie cachee du site : elle recoit les demandes des utilisateurs, verifie les informations, enregistre les donnees et renvoie les bonnes reponses au site.

A dire : Le frontend montre les pages, mais le backend decide si une action est autorisee et garde les donnees coherentes.

1. Le role du backend

  • Il fait le lien entre le site et la base de donnees.
  • Il recoit les actions de l'utilisateur : connexion, recherche, reservation.
  • Il controle que les informations sont correctes avant de les enregistrer.
  • Il renvoie au site les donnees a afficher.
Terme technique : serveur Node.js avec Express
A dire : Une route recoit la demande, le controller traite la logique, puis MySQL renvoie les donnees.

2. Les informations stockees

  • Les comptes clients et les comptes salons.
  • Les informations des salons : ville, adresse, note, description.
  • Les prestations proposees avec leurs prix.
  • Les horaires, les creneaux disponibles et les reservations.
Terme technique : base de donnees MySQL
A dire : Les tables sont reliees entre elles pour savoir quel client reserve quelle prestation, dans quel salon, sur quel creneau.

3. Les comptes utilisateurs

  • Un client peut creer un compte et se connecter.
  • Un salon peut aussi creer un espace professionnel.
  • Les mots de passe ne sont pas stockes en clair.
  • Une fois connecte, le site sait qui utilise l'application.
Terme technique : bcrypt et token JWT
A dire : bcrypt protege les mots de passe, et le JWT permet de reconnaitre l'utilisateur apres la connexion.

4. Les droits d'acces

  • Un client ne peut pas acceder aux outils d'un salon.
  • Un salon ne peut pas reserver comme un client.
  • Certaines actions demandent d'etre connecte.
  • Le backend verifie les droits avant chaque action importante.
Terme technique : middleware d'authentification et de role
A dire : Les droits sont controles cote serveur, car le frontend seul ne suffit jamais pour proteger une application.

5. Recherche des salons

  • L'utilisateur peut chercher un salon par ville.
  • Il peut filtrer selon la note, le prix ou la prestation.
  • Le backend renvoie la liste des salons correspondants.
  • Il permet aussi d'afficher la fiche complete d'un salon.
Terme technique : routes API REST en GET
A dire : Les filtres sont optionnels : la meme route peut chercher par ville, note, prix maximum ou prestation.

6. Reservation d'un rendez-vous

  • Le client choisit un salon, une prestation et un horaire.
  • Le backend verifie que le creneau est encore libre.
  • Si tout est correct, le rendez-vous est enregistre.
  • Le creneau devient ensuite indisponible pour les autres clients.
Terme technique : requete POST et mise a jour SQL
A dire : La reservation utilise une transaction et un verrou SQL pour eviter qu'un meme creneau soit pris deux fois.

7. Espace professionnel

  • Le salon peut gerer ses prestations.
  • Il peut organiser ses horaires et ses creneaux.
  • Il peut consulter les rendez-vous recus.
  • Il garde une vue claire sur son activite.
Terme technique : routes protegees pour le role salon
A dire : Le dashboard est reserve au role salon et centralise la gestion quotidienne.

8. Statistiques

  • Le salon peut voir un chiffre d'affaires simple.
  • Les montants sont calcules a partir des reservations.
  • Le suivi peut se faire sur la journee, la semaine ou le mois.
  • Ces informations aident le salon a suivre son activite.
Terme technique : calculs SQL avec SUM et COUNT
A dire : Les statistiques ne comptent pas les reservations annulees, seulement l'activite confirmee.

Questions jury a anticiper

Pourquoi MySQL ?

Les donnees sont relationnelles : clients, salons, prestations, creneaux et reservations sont lies.

Pourquoi JWT ?

Le token permet au backend de reconnaitre l'utilisateur connecte sur les routes protegees.

Pourquoi une transaction ?

Elle garantit qu'une reservation et l'indisponibilite du creneau sont validees ensemble.

Transition vers la demonstration

  • Montrer une connexion client.
  • Rechercher un salon et ouvrir sa fiche.
  • Reserver un creneau disponible.
  • Passer cote salon pour voir les reservations et les statistiques.

Conclusion

Pour conclure, le backend est la base solide du projet. Il permet aux utilisateurs de se connecter, de chercher un salon, de reserver un rendez-vous, et aux salons de gerer leur activite. Meme s'il ne se voit pas directement, il fait fonctionner toute la logique du site.

A dire : La difficulte principale a ete de garder une logique fiable entre les roles, les reservations et les creneaux disponibles.

Phrase de fin possible

"La plus grande difficulte a ete de bien organiser les droits entre clients et salons, et de s'assurer qu'un creneau reserve ne puisse pas etre pris deux fois."
